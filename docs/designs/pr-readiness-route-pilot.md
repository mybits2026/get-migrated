## Reduced Route Pilot
- Entry point: a dedicated `/pr-readiness` route focused on the temporary graduate visa holder wedge.
- Flow: land on route -> complete triage -> see fit / borderline / out-of-scope result -> if qualified, continue to intake -> submit for manual review -> receive one standardized summary artifact.
- Payment placement: triage happens before any payment decision; payment and partner-flow details remain constrained by the unresolved compliance and operations TODOs.

## Standardized Summary Artifact
- One term only: the standardized summary artifact is the repeatable handoff artifact.
- Contents: user facts, stated assumptions, missing evidence, major uncertainties, next steps, and clear disclaimer boundary.
- Owner: generated by a case reviewer, not auto-generated by the route.
- Operational meaning of `agent-ready`: a licensed reviewer can understand the case shape without re-collecting the same facts from scratch.

## Edge Cases
- Unqualified users are routed away with a clear explanation and no deeper funnel steps.
- Borderline or contradictory answers are flagged for manual review rather than auto-approved.
- Incomplete submissions do not create a complete case.
- If no licensed-help match exists, the artifact still remains useful to the user as an organized self-summary.

## Assumptions And Constraints
- Licensed-help matching remains manual, not automated.
- Provider verification is outside this narrow route-pilot scope.
- Triage uses a simple first-pass rule set, not an open-ended decision engine.
- The route is user-visible; the summary artifact is user-visible and manually produced.

## Success Criteria
- The route clearly filters users into fit, borderline, or out-of-scope buckets.
- At least one repeatable artifact format emerges from real pilot cases.
- The route does not accidentally grow into a built-in workflow product.

## Design Addendum

### Product Type
- This route is a `HYBRID` experience: a marketing/trust-first entry screen that turns into an app-like guided flow.
- The first viewport should obey landing-page composition rules.
- Triage, result, intake, and artifact states should obey calmer app-layout rules.

### Information Hierarchy
- First screen hierarchy:
  1. Trust promise: what this route helps with and what it does not promise.
  2. Target-user cue: temporary graduate visa holders from Nepal already in Australia.
  3. Primary action: `Start PR readiness triage`.
  4. One sentence explaining the three possible outcomes: fit, borderline review, out-of-scope.
- Section map:
  - Hero: qualify the visitor and start the flow.
  - Triage intro: explain what will be asked and how long it takes.
  - Result area: explain fit / borderline / out-of-scope outcome and the next step.
  - Intake area: collect only the facts needed for the reduced pilot.
  - Artifact preview: show what the standardized summary looks like and why it is useful.
  - FAQ / objection handling: answer trust and scope concerns only.

### Emotional Tone
- First-screen tone: clarity-first reassurance, not hype.
- Borderline state should feel like respectful pause and care, not rejection.
- Final artifact should feel like organized next-step confidence, not generic success.

### Layout Model
- Do not default to a card-stack layout for the app-like portions of the route.
- Desktop: split-pane or strong two-zone layout with clear progression and visible context.
- Mobile: stepper-like progression with one main action per step and explicit progress cues.
- Cards should only be used where the card itself is the interaction.

### Interaction States
| Feature | Loading | Empty | Error | Success | Partial |
|--------|---------|-------|-------|---------|---------|
| Triage | Short loading state before showing the result | Not applicable | Contradictory or incomplete answers shown inline with recovery guidance | Fit / borderline / out-of-scope result screen | Step progress visible while answering |
| Borderline result | Review-pending message if manual confirmation is needed | Not applicable | If review cannot proceed, explain why and what the user can do next | Clear waiting state with timeframe and next expectation | Explain that review has not yet started intake |
| Intake | Save/submit loading state with visible progress | Empty initial form with supportive microcopy | Explicit submit-failure state must answer whether the case was saved or not | Confirmation that the case was received | Incomplete form state should preserve progress and error context |
| Artifact | Loading state while artifact is being prepared | No-match state should still present the artifact as a useful self-summary | Artifact failure should not be silent; explain delay or next step | In-page summary shown as the payoff state | Missing evidence appears as clearly labeled gaps, not hidden omissions |

### Outcome Design
- `Fit`:
  - headline explains they are a match for the reduced route pilot
  - CTA moves into intake
- `Borderline`:
  - headline explains a human needs to review before intake continues
  - include timeframe, what happens next, and what the user should do meanwhile
- `Out-of-scope`:
  - headline explains this route is not the right fit
  - provide a respectful explanation and route-away guidance

### Artifact Presentation
- The standardized summary artifact should be experienced in-page first.
- Optional export can be added later.
- The artifact should show:
  - user facts
  - assumptions
  - missing evidence
  - major uncertainties
  - next steps
  - disclaimer boundary

### Motion
- Use only three route-specific motions:
  - subtle hero proof stagger on entry
  - step transition between triage states
  - artifact reveal motion when the summary becomes available
- Avoid decorative or ambient motion beyond those three.

### Responsive And Accessibility
- Mobile should be stepper-like, not a collapsed desktop page.
- Use clear landmarks and heading structure for screen readers.
- After each major transition, move focus intentionally to the new heading or result state.
- Minimum touch targets: 44px.
- Validation errors must be associated clearly with the affected field and summarized accessibly.

### AI Slop Guardrails

Do not use any of the following patterns in this route:
1. Purple/violet/indigo gradient backgrounds or blue-to-purple color schemes.
2. The 3-column feature grid: icon-in-colored-circle + bold title + 2-line description, repeated symmetrically.
3. Icons in colored circles as section decoration (SaaS starter template look).
4. Centered everything (`text-align: center` on all headings, descriptions, cards).
5. Uniform bubbly border-radius on every element (same large radius on everything).
6. Decorative blobs, floating circles, wavy SVG dividers.
7. Emoji as design elements.
8. Colored left-border on cards (`border-left: 3px solid <accent>`).
9. Generic hero copy ("Welcome to...", "Unlock the power of...", "Your all-in-one solution for...").
10. Cookie-cutter section rhythm (hero → 3 features → testimonials → CTA).

Route-specific watch-outs:
- Result screens (fit/borderline/out-of-scope) must not look like marketing outcome cards — use mono data language, not motivational copy.
- The artifact summary should read like a document, not a dashboard widget panel.
- The triage stepper must not feel like a quiz SaaS template — no colored progress bars, no gamification gestures.

### Design System Adaptation

- Preserve the current family from `DESIGN.md`: Fraunces, Instrument Sans, IBM Plex Mono, and the navy/eucalyptus/coral token palette.
- Adapt usage for this route:
  - more procedural hierarchy
  - fewer decorative landing-page gestures
  - stronger use of mono/process language for result and artifact sections
  - trust-first explanation above conversion pressure
